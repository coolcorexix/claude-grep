### 1. [branching] Show HN: Ccgs – Collaborative Claude Code sessions, stored in Git branches
- **Link**: https://news.ycombinator.com/item?id=48426297
- **Platform**: HN
- **Date**: ~June 6, 2026 (7 days ago — at the hard limit, freshest item found today)
- **Context**: Claude Code only. An indie dev built a workaround that stores each Claude Code session transcript in a dedicated git branch (prefix `@ccgs/`) so sessions are version-controlled and diff-able. The entire project exists because Claude Code offers no durable, git-native session persistence or branching UX.
- **Opportunity**: HIGH — active Show HN squarely in the adjacent space; maker-to-maker commentary is explicitly welcome in Show HN threads. This is the freshest pain/opportunity signal found today.

**Draft HN comment**:
> Nice project — git as the storage layer is an interesting call, gets you diff and revert for free. I went a different direction with [claude-grep](https://github.com/coolcorexix/claude-grep): rather than a new storage layer, it reads the existing JSONL session files each agent already writes and surfaces them in a single search TUI. Ctrl-B on any past message row writes a branch in Claude Code's own native fork format (with the `parentSessionId` back-reference), so the branch shows up in `claude --resume` without needing a separate git repo. Also covers OpenCode and Hermes sessions in the same search — cross-agent was the main thing I was missing. Trade-off: my approach is read-only on the storage side; yours owns the lifecycle. Would love to compare notes on how you handle the index-on-startup cost for large session trees.

Completely bypasses the UI picker. Transparent that it's my own tool, but it may bridge the gap while the schema migration is sorted.
