### 1. [BUG] Rewind to here does not revert file changes made by Claude in the same session [branching]
**Link:** https://github.com/anthropics/claude-code/issues/70727 · **Platform:** GitHub (anthropics/claude-code) · **Date:** June 25, 2026 (~16 days ago)

**Context (Claude Code):** A distinct variant of the rewind-is-broken cluster (different from #55347, which was about forks never being created): the UI's "Rewind to here" rolls back conversation history correctly but leaves all file changes on disk intact. Example from the report: asked Claude to add a CSS class to a component — `/rewind` rolls back the chat but the class stays in the file. Never worked correctly (not a regression), labeled `area:tui` + `platform:macOS`. This gap is the code-dimension of the same rewind pain: users can't actually undo a session that went sideways because the file state and the conversation state are decoupled.

**Opportunity:** MEDIUM. Open bug, ~16 days old, no fix scheduled. Demonstrating that claude-grep's Ctrl-B creates a fork-from-past-message — leaving the user on a fresh branch where they can ask Claude to re-run from that point with the git-tracked file state — is a credible workaround answer. Be useful, not promotional: this is a bug report thread.

**Drafted reply (GitHub comment):**
> Seeing the same in v2.1.x — the rewind moves the conversation cursor in the JSONL but doesn't emit any file-restore entries. As a workaround while this is unresolved: Ctrl-B in ccfind / claude-grep (github.com/coolcorexix/claude-grep) forks the session at any past user message and writes a new resumable session file. It doesn't restore the files either, but it does let you fork-from-that-point and re-run Claude against whatever file state git has at that commit — sidesteps the lost-rewind-path problem without relying on the native `/rewind` producing a real fork.
