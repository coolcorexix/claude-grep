### 1. [branching] GitHub #69712: "[FEATURE] Conversation branches (/branch) should surface in agent view, with a merge-back path"
**https://github.com/anthropics/claude-code/issues/69712** · anthropics/claude-code · created June 20, 2026 · open

**Context (Claude Code):** By mid-2026 Claude Code has *three* fork primitives — `/fork` (background subagent, auto-returns), `/branch` (old `/fork` semantics: transcript copy, user-driven independent session), and `--fork-session` (CLI flag). The author's frustration: these are disjoint from `claude agents` (the agent management view). If you branch to explore an alternative, the branch stays invisible in the agent view until you separately background it with `/bg`. More critically: **there is still no `/merge` path** — once you find the winning branch, bringing its conclusions home requires manual export, copy-paste, or git-commit workarounds. The author proposes surfacing `/branch` sessions automatically in `claude agents` and adding `/merge <session>` as a user-triggered pull. References earlier issues #12629 (largely shipped) and #60272 (adjacent).

**What makes this different from #32631 (covered July 2):** #32631 is the canonical spec proposal written in March. #69712 is a *user frustration report* filed after the `/branch` feature shipped — meaning the ask is no longer hypothetical, it's a confirmed gap in a live feature. That makes it a higher-signal comment target: the author has used the feature and hit the wall.

**Opportunity:** HIGH — issue is 13 days old, not yet crowded, and the frustration is specifically about the *missing return path* — which is exactly the angle ccfind's Ctrl-B covers (branch from past message → fork lands in `claude --resume` natively → you can switch back). The `/btw`-compatible note is the credibility line here.

**Drafted reply (GitHub comment):**
> The "no return path" gap you're describing is the piece I felt most after `/branch` shipped. The workflow I landed on in the meantime: use [`ccfind`](https://github.com/coolcorexix/claude-grep) to find the exact message where I wanted to have branched, then hit Ctrl-B to fork the session from that point. The fork writes to Claude Code's native JSONL format (with `parentUuid`/`isSidechain`), so it shows up in `claude --resume` and in the agent view — still no `/merge`, but at least I can switch between the original and the branch without copy-paste. Works across Claude Code, OpenCode, and Hermes sessions.
>
> The native `/merge` and agent-view integration you're spec'ing is the right long-term fix; this is the manual stopgap. (Disclosure: I built ccfind.)

### 2. [branching] GitHub #69272: "[FEATURE] Add /fork (conversation branching) support to VS Code extension"
**https://github.com/anthropics/claude-code/issues/69272** · anthropics/claude-code · created June 18, 2026 · open

**Context (Claude Code — VS Code):** The CLI supports `/fork` and `--fork-session`; the VS Code extension's prompt box does not. Users who work in VS Code can't fork a session without dropping to the terminal. The request mirrors #46451 (covered in prior searches, not in any prior radar header) but was filed separately, suggesting users are still hitting this independently. The distinction between the new `/fork` (background subagent) and `/branch` (clean independent session) adds confusion for extension users who may not track CLI changelog semantics.

**Opportunity:** MED — issue is 15 days old; the VS Code audience is large but ccfind is a terminal TUI so it doesn't directly solve the VS Code friction. A reply that acknowledges the gap and offers a CLI fallback (ccfind for session search + Ctrl-B to fork from search results) is useful but not a slam-dunk. Worth one brief comment if the thread stays uncrowded.

**Drafted reply (GitHub comment):**
> Agreed on the VS Code gap. For the interim: I built a CLI-side fallback — [`ccfind`](https://github.com/coolcorexix/claude-grep) lets you search all your Claude Code sessions by content, then hit Ctrl-B on any result row to fork that session at any past message. The fork appears in `claude --resume` and `claude agents` in the native format, so you can resume from VS Code even if the fork was initiated from the terminal. Not a VS Code extension (yet), but covers the "I want to branch from a point I can find" use case. (Disclosure: I built ccfind.)

### 3. [search] iamironz/opencode-sessions-explorer — second independent OpenCode session-search tool ships June 14
**https://github.com/iamironz/opencode-sessions-explorer** · created June 14, 2026 · 1 ⭐ (early-stage)

**Context (OpenCode):** An OpenCode MCP plugin (17 read-only tools + 1 write tool) that lets you ask natural-language questions about your entire OpenCode session history: "where did I mention X?", "how much did I spend this month?", "which session covered Y?". Operates locally on OpenCode's SQLite database, zero data leaves device. Distinct from joeyism/opencode-history-search (84 ⭐, May 2026 — not in any prior radar either) which does keyword/regex/fuzzy search over the same corpus.

**Why both matter:** Two independent tools built in the same 5-week window (May–June 2026), targeting the same OpenCode search pain from different angles (structured query vs. natural language). This is the clearest demand-validation signal in today's radar — the pain is real enough that multiple makers built the same thing without coordinating.

**What ccfind offers that neither does:** unified search across Claude Code + OpenCode + Hermes in one TUI. Both tools are OpenCode-only and require an MCP setup; ccfind is a standalone CLI.

**Opportunity:** MED-HIGH — the opencode-sessions-explorer README and any future issues are a natural place to leave a "complementary tool" note. The iamironz repo is 19 days old and has 0 issues, suggesting the author is still actively checking it. A respectful "I built something adjacent" note on the repo (via a GitHub issue or README PR) reaches someone who clearly understands the problem deeply.

**Drafted GitHub issue (titled: "Cross-agent search as a complement?"):**
> Great tool — built for the same pain from a different angle. I made [`ccfind`](https://github.com/coolcorexix/claude-grep) (a standalone TUI) that searches Claude Code, OpenCode, and Hermes sessions from one interface without needing MCP. Different trade-offs: no natural-language query, but zero setup and cross-agent coverage. Thought it might be useful context for your roadmap or for users who split time across agents. Happy to swap notes on what you've learned about the SQLite schema quirks. (Disclosure: I built ccfind.)
