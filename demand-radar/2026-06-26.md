### 1. [search] GitHub issue #71473 — June 25, 2026 (last night) ⭐ FRESH
**"Code/SSH: all session history blank when ~/.claude/projects is a symlink (transcript locator runs `find` without -L)"**
[https://github.com/anthropics/claude-code/issues/71473](https://github.com/anthropics/claude-code/issues/71473) · anthropics/claude-code · open, 0 comments

**Context (Claude Code):** User @mdavies has a dotfiles/synced-git setup where `~/.claude/projects` is a symlink — common for anyone who version-controls or syncs their home directory. The Claude Desktop Code tab shows "No messages yet" for every session, even though JSONL files are intact on disk and `claude --resume <id>` works fine from the CLI. Root cause: the Desktop's transcript locator runs `find ~/.claude/projects -name '<id>.jsonl'` without `-L`; GNU `find` refuses to descend into a symlinked starting directory in default `-P` mode, finds nothing, returns exit 0, and the conversation pane stays blank for every session. The user has a working workaround (bind-mount) but notes "users should never need to touch internal state."

**Opportunity:** HIGH. Pure "data is on disk, official UI can't reach it" scenario — the sharpest fit in the queue today. `ccfind` reads the JSONL files directly via standard `open()` / path resolution, which follows symlinks without any special flag. Works regardless of whether `~/.claude/projects` is a real directory or a symlink.

**Drafted reply (GitHub comment):**
> `ccfind` (from claude-grep) works around this right now — it reads the JSONL files directly via standard `open()`, which follows symlinks, so the missing `-L` in the `find` locator doesn't affect it. While waiting for the harness fix you can still search and access all your session content. `brew install coolcorexix/tap/claude-grep` or `nix run github:coolcorexix/claude-grep`. (Disclosure: I'm the author of claude-grep.)
