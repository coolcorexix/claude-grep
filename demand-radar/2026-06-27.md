### 2. [branching] GitHub issue #70727 — June 25, 2026 (2 days ago, not in prior radars)
**"[BUG] Rewind to here does not revert file changes made by Claude in the same session"**
[https://github.com/anthropics/claude-code/issues/70727](https://github.com/anthropics/claude-code/issues/70727) · anthropics/claude-code · open

**Context (Claude Code):** Reporter @ZenaraJaya shows a minimal repro: ask Claude to make a file change, then click "Rewind to here" on the message before the request. The conversation history rewinds correctly — but the files on disk stay modified. The reporter notes "this feature never worked" and labels it platform:macos, area:tui. This is distinct from issue #55347 (rewind doesn't create a resumable fork) — here the coupling assumption goes the other way: rewind was supposed to roll back *both* conversation and files, but only rolls back conversation.

**Opportunity:** HIGH — thread is 2 days old with no comment yet (0 comments), perfect window. The underlying insight is that conversation-state and file-state are fundamentally different axes: branching the conversation is cheap (rewrite a JSONL), reverting files requires something like git. claude-grep's Ctrl-B forks the session at the conversation level only, which is what most people actually want when they "go back to an earlier message."

**Drafted reply (GitHub comment):**
> This exposes the fundamental mismatch in `/rewind`'s design: conversation history and file state are two different things. Rolling back the conversation is cheap (it's just JSON); rolling back files cleanly needs git semantics. Conflating the two is what makes the feature so brittle — either the conversation rewinds without the files (#70727, here) or the fork promised never materialises (#55347).
>
> If your underlying goal is "branch the conversation from this earlier message and try a different approach," claude-grep ships a Ctrl-B prompt-tree feature that does exactly that: forks a new Claude Code session at any past user message, leaving the original session intact and producing a branch that shows up in `/resume`. No file-state assumptions. (Disclosure: I'm the author.) `brew install coolcorexix/tap/claude-grep`

### 3. [branching] GitHub issue #55347 — May 1, 2026 (standing open, 8 weeks)
**"/rewind says 'conversation will be forked' but never creates a resumable fork"**
[https://github.com/anthropics/claude-code/issues/55347](https://github.com/anthropics/claude-code/issues/55347) · anthropics/claude-code · open

**Context (Claude Code):** `/rewind` promises "The conversation will be forked" but mutates the session in place — the original session vanishes from `--resume` and no new session is created. Filed May 1, still unresolved. Together with #70727 above, this forms a two-bug cluster showing the rewind/fork machinery is unreliable on both axes. The volume of related feature requests (#32631 full branching spec, #16276, #16236, #46451 VS Code extension fork) confirms the demand is structural, not niche.

**Opportunity:** MED (8 weeks old, but still open and gets new "me too" comments). Worth a reply if there's fresh activity in the thread today; otherwise hold until the bug is fixed/acknowledged.

**Drafted reply (GitHub comment, hold for fresh activity):**
> The root issue is that `/rewind` rewrites the branch pointer in the same JSONL file rather than spawning a new session ID — so there's no fork object for `/resume` to surface. As a stopgap: claude-grep's Ctrl-B does create a new session JSONL (with its own ID) branched at a past message, `/btw`-compatible so it shows up in `claude --resume` normally. (Disclosure: my project.) `brew install coolcorexix/tap/claude-grep`
