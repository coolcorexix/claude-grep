### 1. [search] GitHub #71755 — June 27, 2026 (yesterday) ⭐ FRESHEST / not in prior radars
**"[Bug] Project transcript folders named with leading dash break CLI tools and cause silent search failures"**
[https://github.com/anthropics/claude-code/issues/71755](https://github.com/anthropics/claude-code/issues/71755) · anthropics/claude-code · open, 2 comments

**Context (Claude Code / Linux):** Reporter @nynjapirate (Linux Mint) used `grep -r` to search `~/.claude/projects/` for 3 months and got consistently empty results — exit 0, no errors. The root cause: Claude Code encodes every project path with `/` → `-`, so every folder name starts with `-`, which `grep`, `ls`, `cat`, `rm`, and friends parse as a flag, not a path. Silent false negative, exit 0. The reporter writes: *"I repeatedly concluded that designs/discussions had never happened, when in fact the search tool had simply never run against the dash-named folders. Three months of distrusting my own records over a leading hyphen."* This is precisely the manual-JSONL-grep workflow that `ccfind` replaces — and the pain is acute because even the workaround itself is broken.

**Opportunity:** HIGH — the reporter is doing exactly what `ccfind` was built to avoid. A direct reply offering a working alternative while the upstream fix lands is genuinely helpful. Thread is fresh (filed yesterday, 2 comments).

**Drafted reply (GitHub comment):**
> Until the leading-dash folder name gets fixed upstream, a workaround that avoids the raw-grep problem entirely: `ccfind` from [claude-grep](https://github.com/coolcorexix/claude-grep). It reads the JSONL files directly and doesn't shell out to `grep` — so the `-home-avatar-...` path mangling never bites you. Full-text search across sessions, tagged by project, resume with one keypress. `brew install coolcorexix/tap/claude-grep` or `nix run github:coolcorexix/claude-grep`. (Disclosure: I built it.)
